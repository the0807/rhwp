# Task M100 #949 Stage 25 계획서 — `hwpx-h-03` GenShape CTRL_HEADER 계약 해석

## 1. 배경

Stage 24에서 `hwpx-h-03` 파일손상 지점을 다음 record 경계로 고정했다.

```text
drawText/image group tuple : #820..#838
following table paragraph : #839..#858
```

확인된 사실:

```text
1. drawText 내부 LIST_HEADER/PARA_HEADER는 generated와 oracle이 일치한다.
2. hp:pic@href -> CTRL_DATA payload/위치도 #833에서 oracle과 일치한다.
3. following table paragraph의 PARA_TEXT #840, TABLE CTRL_HEADER #843, TABLE #844도 일치한다.
4. 남은 강한 차이는 #824/#831/#835 GenShape CTRL_HEADER와 관련 SHAPE_COMPONENT/SHAPE_PICTURE payload다.
```

따라서 Stage 25는 TextBox envelope 전체를 다시 보강하지 않는다. `GenShape CTRL_HEADER`
계약을 field 단위로 해석하고, HWPX source 속성이 HWP5 record 값으로 어떻게 내려가야 하는지
규칙을 세운다.

## 2. 목표

`hwpx-h-03`의 파일손상 후보를 다음 세 record의 구체 field 차이로 설명한다.

```text
#824 outer rect CTRL_HEADER
#831 first child picture CTRL_HEADER
#835 second child picture CTRL_HEADER
```

이번 단계의 목적은 구현이 아니라 field decode와 적용 조건 확정이다.

## 3. 작업 원칙

```text
1. Stage 23의 generic TextBox LIST/PARA header 보강은 재사용하지 않는다.
2. `hwpx-h-01`, `hwpx-h-02`에서 통과한 table-axis 계약은 건드리지 않는다.
3. `hy-001` #974 guard와 충돌하는 일반 규칙은 후보에서 제외한다.
4. source HWPX 속성, oracle HWP5 payload, generated HWP5 payload를 3자 비교한다.
5. payload hash 차이만으로 결론 내리지 않고 byte offset과 의미 필드 후보를 적는다.
```

## 5. 절차

### 5.1 CTRL_HEADER payload byte diff 작성

대상 record:

```text
#824 CTRL_HEADER
#831 CTRL_HEADER
#835 CTRL_HEADER
```

확인할 내용:

```text
1. oracle/generated byte offset diff
2. 4바이트 단위 little-endian 해석
3. control id, attr, position, size, margin, z-order 후보 field
4. source XML 속성과 대응 가능한 field
```

산출물:

```text
output/poc/hwpx2hwp/task949/stage25_genshape_ctrl_header_decode/h03_ctrl_header_field_diff.md
```

### 5.2 source 속성 대응표 작성

`hwpx-h-03` source XML에서 target shape/picture의 속성을 뽑아 HWP5 후보 field와 대응한다.

중점 속성:

```text
hp:rect@textWrap
hp:rect@textFlow
hp:rect@groupLevel
hp:rect@instid
hp:drawText@lastWidth
hp:pic@href
hp:pic@instid
hp:pic의 position/size 계열 속성
```

산출물:

```text
output/poc/hwpx2hwp/task949/stage25_genshape_ctrl_header_decode/h03_source_to_hwp5_mapping.md
```

### 5.3 guard 샘플과 적용 조건 비교

`hwpx-h-01`, `hwpx-h-02`, `hy-001`에서 같은 규칙을 적용해도 되는지 조건을 분리한다.

확인할 내용:

```text
1. target과 같은 drawText 내부 picture pair 구조인지
2. textWrap/textFlow 조합이 같은지
3. child picture가 href 있는 그림과 href 없는 그림의 조합인지
4. 한컴 oracle에서 같은 attr high bits가 나타나는지
5. #974 hy-001 guard의 TAC space 조판과 충돌하는지
```

산출물:

```text
output/poc/hwpx2hwp/task949/stage25_genshape_ctrl_header_decode/guard_applicability.md
```

### 5.4 다음 구현 후보 판정

Stage 25 끝에서 다음 중 하나로 판정한다.

```text
A. #824/#831/#835 CTRL_HEADER field mapping만으로 Stage 26 구현 후보를 만들 수 있다.
B. CTRL_HEADER만으로 부족하고 SHAPE_COMPONENT/SHAPE_PICTURE payload decode가 먼저 필요하다.
C. 현재 도구로는 의미 field를 식별하기 부족하므로 hwp5-inventory decode 기능을 보강해야 한다.
```
